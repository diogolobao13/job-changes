# job-change-detector

Detecta mudança de empresa de contatos do Active. A busca da Unipile só traz a URL,
então: casa a URL contra `active_campaign_linkedin` (por slug), e só pra quem está na
base faz a 2ª request (retrieve profile) pra pegar a empresa atual. A linha do contato
é **sempre atualizada com a última empresa pesquisada**. Quando a empresa do fetch
difere da que está na base, grava em `linkedin_job_changes` (só a empresa atual) e
dispara o webhook. Roda como GitHub Action (cron).

## 1. Supabase

`active_campaign_linkedin` já existe. Criar a tabela de mudanças (sem old/new — só a
empresa atual) e um índice pra acelerar o match:

```sql
create table if not exists public.linkedin_job_changes (
  id           bigint generated by default as identity primary key,
  contact_id   bigint not null,
  linkedin_url text,
  full_name    text,
  company      text,
  detected_at  timestamptz not null default now()
);

create index if not exists idx_aclinkedin_url
  on public.active_campaign_linkedin (linkedin_url);
```

## 3. Lógica de match e comparação

- Match por **slug** do `/in/<slug>` da URL (ignora `http/https`, `www`, barra final, query).
- Não está no Active → ignora (não gasta fetch de perfil).
- Está no Active → busca empresa atual e:
  - **atualiza sempre** a linha com a última empresa pesquisada (`company_name`,
    `last_update_date`, `updated_at`, e `uuid_linkedin` se estiver vazio);
  - dispara webhook **só** quando já havia `company_name` e ela mudou.

## 4. Payload do webhook (uma chamada por mudança)

```json
{
  "contact_id": 123,
  "linkedin_url": "https://www.linkedin.com/in/fulano-silva",
  "full_name": "Fulano Silva",
  "company": "Empresa Atual",
  "detected_at": "2026-06-03T09:17:00.000Z"
}
```

## Notas

- **Rate limit**: retrieve profile é limitado (~150/dia em Sales Navigator).
  `MAX_PROFILE_FETCHES=120` e `PROFILE_DELAY_MS=4000` mantêm margem; o que passar do
  teto fica pra próxima rodada.
- **Cron best-effort**: pode atrasar/pular; faça ≥1 commit a cada ~60 dias pra não
  desativar o agendamento por inatividade.
- Os dois `AJUSTE` no `src/index.ts` (campo da URL na busca e extração da empresa no
  perfil) devem ser confirmados com 1 payload real — `console.log(JSON.stringify(...))`.
